# Введение

Мне вдруг взбрело в голову сделать небольшую игру на своём лендинге — пиксель-баттл. Честно говоря, идея - чисто фана ради, показать свои навыки и немного занять голову на пару вечеров. Запасся ящиком энергетиков и накодил что-то.

Для проекта я взял `socket.io` для реалтайма, `postgres` как БД и `telegram` для авторизации.

Схема и логика очень просты: сервер хранит текущие координаты и цвет пикселей, при подключении шлёт инициализацию `init`, а при рисовании — обновления `draw` / `update`.

# Схема базы и ранняя ошибка

Изначально я хранил пиксель как оффсет (целое число):

```sql
CREATE TABLE "canvas" (
	"offset" integer PRIMARY KEY NOT NULL,
	"color" text,
	"updated_at" timestamp DEFAULT now(),
	"updated_by" uuid,
);
```

`offset` — индекс клетки на полотне, `color` — hex-цвет. Всё работало, пока я не поменял размер полотна — тогда offset перестал работать и надо было поменять модель.

Я исправил модель, заменив `offset` на нормальные координаты:

```sql
CREATE TABLE "canvas" (
	"coordinates" text PRIMARY KEY NOT NULL,
	"color" text,
	"updated_at" timestamp DEFAULT now(),
	"updated_by" uuid,
	CONSTRAINT "canvas_coordinates_unique" UNIQUE("coordinates")
);
```

# События socket.io (пример)

Я прописал ивенты для socket.io: апдейты пикселей, данные о пользователе и самое важное: инициализация полотна при 1 загрузке

```ts
socket.on('pb:init', async () => {
    const result = await (await db.select({ i: canvas.coordinates, c: canvas.color, u: canvas.updated_by, a: canvas.updated_at }).from(canvas)).map(v => {
        const [ x, y ] = v.i.split(':').map(Number) as [number, number]
        return { ...v, i: { x, y } }
    })
    socket.emit('pb:init:response', result)
})
socket.on('pb:draw', async (data: { color: number, coordinates: { x: number, y: number }, user: TelegramAuthUser }) => {
    const result = await db.insert(canvas).values({ color: data.color, coordinates: `${data.coordinates.x}:${data.coordinates.y}`, updated_at: new Date(), updated_by: data.user.uuid }).onConflictDoUpdate({
        target: canvas.coordinates,
        set: {
            color: data.color, updated_at: new Date(), updated_by: data.user.uuid
        }
    }).returning()
    const [ x, y ] = result[0].coordinates.split(':').map(Number) as [number, number]
    socket.broadcast.emit('pb:update', { color: result[0].color, coordinates: { x, y } })
    await db.update(user).set({ online: true }); debouncer.use(async () => await db.update(user).set({ online: false }), data.user.uuid)            
})
```

# Проблемы при нагрузочном тестировании

Пока я тестировал с друзьями, сначала всё шло весело: пиксели летели, обновления работали. 

1. `pb:init` — Payload вырос из нескольких килобайт в >90 МБ. Я думаю не нужно объяснять что это мягко говоря – много.
2. `pb:draw` — каждое действие пользователя отправлялось как отдельное сообщение, что приводило к спаму в сокете и плохой масштабируемости.

Формат, который использовался изначально — "голый" JSON-массив — оказался неприемлемым.

# Первые оптимизации: убрать лишнее и индексация цветов

Исходное представление одного пикселя выглядело так:

```js
[
  { i: { x: 10, y: 20 }, c: '#ff0000', u: 'user_name', a: '2025-08-18T12:00:00Z' },
  ...
]
```

Чтобы сэкономить место сначала сделал самое очевидное:

1. Выкинул несущественные поля (например `u`, `a` — они не нужны в init-пакете).
2. Убрал нулевой (чёрный) цвет, хотя эффект был небольшой т.к. черным цветом рисуют редко.
3. Представил палитру цветов числовыми индексами:

```js
{ 
    0: '#000000', 
    1: '#ffffff', 
    2: '#f97316', 
    3: '#eab308', 
    4: '#10b981', 
    5: '#2B7FFF', 
    6: '#a855f7', 
    7: '#ec4899', 
    8: '#e11d48', 
    9: '#FFC79F' 
}
```

Теперь запись стала компактнее:

```js
[
  { x: 10, y: 20, c: 1 },
  ...
]
```

Уже лучше, видна структура и возможность абстракции, посмотрев на этот обьект, первая мысль которая приходит – превратить его в массив и сериализовать.

# Текстовая сериализация

Для начала я попробовал просто текстом:

```
10:20:1;10:21:1;10:22:1;...;1024:1024:9;
```

Это уже короче, но всё равно слишком много лишних символов. Нужно было что-то эффективнее.

# Bit-пакетирование (bit-packing)

Бродив по просторам интернета я наткнулся на данный способ сжатия нескольких чисел. Если очень просто, то: 

**Bit-пакетирование** — это когда ты берёшь несколько маленьких чисел,
сдвигаешь их на нужное количество бит и складываешь в одно число,
а потом можешь точно так же «распаковать» их обратно с помощью сдвигов и масок.

Для упаковки координат и цвета надо прописать им диапазоны и во сколько байт они вместятся:

* `x` ∈ `0–1023` → 10 бит
* `y` ∈ `0–1023` → 10 бит
* `c` ∈ `0–9` → 4 бита (т.к. 2⁴ = 16 > 9)

Простая упаковочная функция:

```js
const pack = (x, y, c) => (x << 14) | (y << 4) | c!
```

Таблица (на пальцах):

| Поле | Размер | Пример значения | Бинарное представление |
| ---- | ------ | --------------- | ---------------------- |
| `x`  | 10 бит | `512`           | `10 0000 0000`         |
| `y`  | 10 бит | `321`           | `0101 0000 01`         |
| `c`  | 4 бит  | `7`             | `0111`                 |

Объединяя эти части, получаем последовательность бит, которую можно записать в 3 байта (24 бит).

```
[00001000000000][0101000001][0111]
 ^x(10 бит)^     ^y(10)^    ^c(4)^
```

Идеально, лучше сжать не получится

# Передача: строки vs бинарные фреймы

Первой мыслью было превратить каждый packed-int в строку (base64/base64-подобные), с одной стороны – логично, 3 байта в представлении base64 дают ровно 4 символа (4 байта)

```ts
const result = data.filter(v => v.c !== 0).map(({ x, y, c }) => ((x << 14) | (y << 4) | c!).toString(64).padStart(4,'0')).join('')
```

Оказалось – WebSocket поддерживает не только текстовые сообщения `opcode 0x1`. но и бинарные фреймы `opcode 0x2`, поэтому лучше отправлять raw-байты.

Решение пришло само собой: сформировать `Uint8Array`, записать в него 24-битные подрядные записи и отправить как бинарный фрейм.

```ts
const pack = async (data: { x: number, y: number, c: number }[]) => {
    const buffer = new Uint8Array(Math.ceil(data.length * 24 / 8));
    data.forEach(({i, c}, idx) => {
        for (let b = 0; b < 24; b++) {
            const bit = (((x << 14) | (y << 4) | c!) >> (23 - b)) & 1;
            const index = idx * 24 + b;
            buffer[Math.floor(index / 8)] |= bit << (7 - (index % 8));
        }
    });
    return buffer
}
```

И отправляем при инициализации:

```ts
socket.emit('pb:init:response', pack(data));
```

# Распаковка на клиенте

Клиент разбирает бинарный буфер, читает 24-битные блоки и смещает значения обратно:

```ts
unpack: (buffer: Uint8Array | ArrayBuffer) => {
    if (buffer instanceof ArrayBuffer) buffer = new Uint8Array(buffer);
    const array = [];
    for (let n = 0; n < Math.floor((buffer.length * 8) / 24); n++) {
        let num = 0;
        for (let b = 0; b < 24; b++) {
            const index = n * 24 + b;
            num = (num << 1) | ((buffer[Math.floor(index / 8)]! >> (7 - (index % 8))) & 1);
        }
        array.push({
            x: (num >> 14) & 0x3FF, // 10 бит
            y: (num >> 4) & 0x3FF, // 10 бит
            c: num & 0xF // 4 бит
        });
    }
    return array;
},
```

# Буферизованные апдейты

Завершающий штрих, чтобы не спамить сокет при каждом обновлении каждого пикселя, добавил на сервере стек и периодически отправляю батч всем участникам:

```ts
let stack: { x: number, y: number, c: number }[] = []
setInterval(() => {
    if (!stack.length) return
    socket.broadcast.emit('pb:update', pack(stack))
    stack = []
}, 2000)
```

# Результат

* Каждый пиксель теперь занимает 24 бита вместо 80+ байтов.
* Init-пакет стал маленьким и быстрым (10ms compute вместо 30~ секунд).
* Буферизация апдейтов сгладила нагрузку на мой бедный сервер и его CPU-time.

# Заключение

Bit-пакетирование — это простой и эффективный способ сжатия чисел. Теоретически на готовом бандле можно применять дополнительные алгоритмы сжатия, но на практике это почти не имеет смысла: размер данных уже очень мал, а время декодирования будет непропорционально большим для получения всего около 150 КБ.

В целом эксперимент оказался удачным и в какой то мере даже веселым.
